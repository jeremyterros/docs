## Prérequis

- Serenest confirme que les Webviews du site sont activées.
- Les deux parties s'accordent sur le flux utilisateur :
  - Pour un utilisateur déjà présent dans les deux systèmes : connexion automatique à Serenest.
  - Pour un nouvel utilisateur : provisionnement automatique du compte Serenest via les attributs SAML.

## Étapes de mise en place

<Steps>
  <Step title="Serenest fournit les informations SP">
    Serenest communique au tiers :
    - **ACS URL** : `https://api.serenest.fr/saml/acs/{siteUuid}`
    - **SP Metadata URL** : `https://api.serenest.fr/saml/metadata/{siteUuid}`
    - **SP Entity ID** : `serenest-sp` (ou personnalisé si configuré)
  </Step>
  <Step title="Le tiers fournit ses informations IdP">
    Le tiers communique à Serenest :
    - **IdP Entity ID** (identifiant SAML issuer)
    - **IdP SSO URL** (endpoint de login SAML)
    - **Certificat X.509** de signature au format PEM (fichier `.pem`)
  </Step>
  <Step title="Configuration back-office Serenest">
    Serenest configure l'intégration SAML dans le back-office admin en utilisant les informations fournies par le tiers.
  </Step>
  <Step title="Configuration côté IdP">
    Le tiers configure son IdP avec :
    - L'**ACS URL** fournie par Serenest
    - Le **SP Entity ID** (`serenest-sp` ou tel que spécifié)
    - Les **mappings d'attributs** : `emailaddress`, `givenname`, `surname`, `name`, `badgenumber`
  </Step>
  <Step title="Tests en pré-production">
    Les deux parties testent l'intégration :
    - Serenest crée un compte de test pour le tiers.
    - Le tiers provisionne un compte de test dans son système pour Serenest.
    - Vérification du flux de connexion de bout en bout en pré-production.
  </Step>
  <Step title="Mise en production">
    Une fois le flux validé en pré-production, la configuration est répliquée en production.
  </Step>
</Steps>

## Exemple de requête SAML

Lorsqu'un utilisateur se connecte via l'IdP, celui-ci envoie une requête **POST** vers l'ACS URL :

```
POST https://api.serenest.fr/saml/acs/{siteUuid}
Content-Type: application/x-www-form-urlencoded

SAMLResponse=PHNhbWxwOlJlc3BvbnNlIC...base64...
```

La `SAMLResponse` est un document XML signé (encodé en Base64) contenant les attributs de l'utilisateur. Une fois décodé et validé, Serenest extrait les attributs suivants :

| Attribut SAML    | Description                                        |
| ---------------- | -------------------------------------------------- |
| `emailaddress`   | Email de l'utilisateur                             |
| `givenname`      | Prénom                                             |
| `surname`        | Nom de famille                                     |
| `name`           | Nom complet (username)                             |
| `badgenumber`    | Numéro de badge (peut déjà exister côté Serenest)  |

> **Note :** Ces noms d'attributs sont les valeurs par défaut. Si l'IdP du tiers utilise des noms différents (ex: `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress`), Serenest peut configurer un mapping personnalisé dans les options avancées du back-office.

## Tests fonctionnels

Une fois la configuration technique terminée, les tests suivants doivent être effectués :

| Test                        | Description                                                        |
| --------------------------- | ------------------------------------------------------------------ |
| **Connexion automatique SSO**   | Vérifier que l'utilisateur est connecté automatiquement via le SSO |
| **Rechargement de badge**       | Tester avec un montant réel et un montant de 0                     |
| **Webview**                     | Vérifier l'ouverture transparente sans connexion manuelle          |
| **Nouvel utilisateur**          | Vérifier le provisionnement automatique du compte                  |
