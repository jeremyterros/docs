## Comment sait-on à quel site appartient un badge ?

Le badge est lié à un site spécifique de plusieurs manières :

- **Lors de l'inscription manuelle (sans SSO)** : l'application vérifie que le badge saisi correspond au site mentionné. L'utilisateur est ensuite automatiquement lié à l'entreprise associée à ce site.
- **Via le SSO** : lorsqu'un utilisateur se connecte via SSO, les informations de site et d'entreprise sont fournies par le fournisseur d'identité (IdP) et utilisées pour associer automatiquement le badge au bon site.
- **Lors de l'ouverture d'un nouveau site Innovorder** : un badge de test est fourni par Innovorder et est spécifiquement lié au nouveau site pendant les phases de test.

<Info>
  En résumé : le badge est toujours associé à un site via un import, une vérification lors de l'inscription ou via les données SSO.
</Info>

## Faut-il un email pour s'inscrire à l'application Serenest ou le badge suffit-il ?

Cela dépend du contexte :

### Avec SSO activé

- Si l'utilisateur a un **compte PI Electronique avec email** : le compte est récupéré par email et le badge est mis à jour.
- S'il a un **compte PI avec badge mais sans email** : le compte est récupéré via le numéro de badge.
- S'il **n'a pas de compte Serenest** : un nouveau compte est créé avec les informations fournies par le SSO. **L'email n'est pas obligatoire** dans ce cas.

### Sans SSO (inscription manuelle)

L'utilisateur doit fournir :
- Numéro de badge *
- Prénom *
- Nom *
- Email *
- Mot de passe *

L'**email est obligatoire** dans le processus d'inscription manuelle.

### Récapitulatif

| Contexte | Email obligatoire ? |
| --- | --- |
| Inscription manuelle (sans SSO) | **Oui** |
| Connexion SSO (compte existant ou nouveau) | **Non** — les informations viennent du SSO |
| Compte PI avec badge mais sans email | **Non** — le badge suffit |

## Mon client souhaite mettre à jour tous les badges. Comment procéder ?

<Steps>
  <Step title="Mettre à jour les badges chez le fournisseur">
    Mettre à jour les badges d'abord dans le système du fournisseur (PI Electronique / Innovorder). Serenest récupère toujours les données de badge depuis le fournisseur. Les utilisateurs doivent exister avec leur *nouveau* badge.
  </Step>
  <Step title="S'assurer que les emails restent les mêmes">
    La continuité de connexion dépend de la correspondance par email quand c'est possible. Si les emails ne changent pas, Serenest liera automatiquement le badge mis à jour lors de la connexion.
  </Step>
  <Step title="Pour PI Electronique">
    Serenest mettra à jour le badge automatiquement lors de la connexion si le compte PI peut être trouvé **par email ou par ancien numéro de badge**. Si le compte PI n'a **pas d'email**, Serenest peut quand même faire la correspondance par badge. Les nouveaux numéros de badge doivent déjà exister dans PI avant la connexion des utilisateurs.
  </Step>
  <Step title="Pour Innovorder">
    Envoyer le fichier utilisateurs mis à jour (prénom, nom, emails, *nouveaux badges*) pour import. Innovorder l'importe, puis Serenest le récupère. Une fois importé, les utilisateurs peuvent se connecter normalement avec leur nouveau badge.
  </Step>
  <Step title="Si le SSO est activé">
    L'IdP doit commencer à envoyer les *nouveaux numéros de badge* dans le payload SAML. Serenest se synchronisera automatiquement.
  </Step>
</Steps>

<Tip>
  **En résumé :** Mettre à jour tous les badges au niveau du fournisseur → s'assurer que les emails restent inchangés → s'assurer que le SSO envoie les nouveaux badges → Serenest réassociera automatiquement les comptes lors de la prochaine connexion. Cela garantit **zéro impact** et **toutes les fonctionnalités**.
</Tip>

<Note>
  Aucune mise à jour de masse n'est nécessaire côté Serenest. L'application met à jour les informations utilisateur dynamiquement lors de la connexion (SSO ou classique). Rien ne casse pour les utilisateurs tant que les données en amont sont correctes.
</Note>
